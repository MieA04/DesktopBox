### 架构层次职责

| 层级 | 职责 | 对应需求 |
|------|------|----------|
| **核心框架层** | 模块容器、拖拽/缩放/贴靠引擎、事件总线、状态管理、持久化 | REQ-FRM-001~006, REQ-PST-001~002 |
| **模块层** | 各模块业务逻辑与渲染 | REQ-ICON-001~007, REQ-MON-001~005, REQ-PRO-001~004, REQ-TERM-001~006 |

| **命令层** | Tauri invoke 入口，参数校验与路由 | 全部命令式需求 |
| **服务层** | 后台数据采集、系统 API 封装、定时推送 | 全部数据采集需求 |

## 二、前端架构设计

### 2.1 核心框架层 (Core) — 对应 REQ-FRM-001~006

#### 2.1.1 ModuleBase — 模块基类（模板方法模式）

```typescript
// 所有模块的统一基类，定义模块生命周期
abstract class ModuleBase {
  // 受保护属性
  protected id: string;                    // 模块唯一标识
  protected title: string;                 // 模块标题
  protected container: HTMLElement;        // 主容器 DOM
  protected contentArea: HTMLElement;      // 内容区 DOM
  protected state: ModuleState;            // 位置/尺寸/停靠状态

  // 生命周期方法（由 ModuleManager 调用）
  abstract init(): void;                   // 模块初始化，创建 DOM 内容
  abstract destroy(): void;                // 模块销毁，清理资源

  // 子类可覆写的渲染钩子（模板方法）
  protected abstract renderContent(): void; // 填充 contentArea
  protected onActivate(): void {}           // 模块被激活（REQ-FRM-004）
  protected onDeactivate(): void {}         // 模块失去焦点
  protected onDockChange(dock: DockState): void {}  // 停靠状态变化（REQ-FRM-003）

  // 基类实现的公共方法
  setPosition(pos: Position): void;         // 更新位置 → 持久化（REQ-FRM-001/005）
  setSize(size: Size): void;                // 更新尺寸 → 持久化（REQ-FRM-002/005）
  setZIndex(z: number): void;               // 更新层级（REQ-FRM-004）
  getState(): ModuleState;                  // 获取完整状态
  show(): void;                             // 显示
  hide(): void;                             // 隐藏

  // 亚克力材质控制（REQ-FRM-006）
  setBlurStrength(px: number): void;        // 设置模糊强度
  setBgOpacity(opacity: number): void;      // 设置底色透明度
}
```

**REQ 覆盖：** REQ-FRM-001（拖拽）、REQ-FRM-002（缩放）、REQ-FRM-003（贴靠）、REQ-FRM-004（层级）、REQ-FRM-005（持久化）、REQ-FRM-006（亚克力材质）

#### 2.1.2 ModuleManager — 模块管理器（单例 + 注册模式）


#### 2.1.3 DragEngine — 拖拽引擎（策略模式）

```typescript
// 拖拽行为策略接口
interface DragStrategy {
  onStart(e: PointerEvent, module: ModuleBase): void;
  onMove(e: PointerEvent, module: ModuleBase): void;
  onEnd(e: PointerEvent, module: ModuleBase): void;
}

// 三种具体策略
class FreeDragStrategy implements DragStrategy {
  // [REQ-FRM-001] 自由悬浮拖拽
  // 直接跟随鼠标移动，使用 transform: translate()
  // 使用 PointerEvent 统一鼠标/触控/笔
  // 拖拽期间 pointer-events: none 防干扰
}

class SnapDragStrategy implements DragStrategy {
  // [REQ-FRM-003] 贴靠模式解锁后的受限拖拽
  // 检测边缘阈值（50px），吸附到屏幕边缘
}

class ResizeStrategy implements DragStrategy {
  // [REQ-FRM-002] 缩放拖拽
  // 从边缘/角落拖拽改变 width/height
  // 最小尺寸限制 150×100 px
}

// 拖拽引擎主类
class DragEngine {
  private strategy: DragStrategy;

  setStrategy(strategy: DragStrategy): void;
  attach(module: ModuleBase): void;        // 为模块绑定拖拽事件
  detach(module: ModuleBase): void;        // 解绑
}
```

**核心实现要点：**
- 使用 `PointerEvent` 而非 `MouseEvent`（统一鼠标/触控/笔）
- 移动使用 `transform: translate()` 而非修改 `left/top`（GPU 合成层，不触发重排）
- 拖拽期间设置 `pointer-events: none` 防止 iframe 等子元素干扰
- 结束拖拽时判定是否进入贴靠模式（阈值 50px）
- 贴靠模式下禁止拖拽，需双击标题栏解锁

#### 2.1.5 StateManager / AppState — 全局状态管理（信号模式）

使用 `@preact/signals` 实现轻量响应式状态。

```typescript
import { signal, computed } from '@preact/signals';

class AppState {
  // ── 模块状态 ──
  modules = signal<ModuleData[]>([]);             // 全部模块的状态快照
  activeModuleId = signal<string | null>(null);    // 当前激活模块 ID

  // ── 系统数据（Rust 定时推送）──
  systemStats = signal<SystemStats | null>(null);  // [REQ-MON-001~003] CPU/GPU/内存
  processList = signal<ProcessInfo[]>([]);         // [REQ-PRO-001] 进程列表

  // ── 配置 ──
  settings = signal<AppSettings>(DEFAULT_SETTINGS); // [REQ-CFG-001~003]

  // ── 计算信号 ──
  activeModule = computed(() =>
    this.modules.value.find(m => m.id === this.activeModuleId.value)
  );
  memoryPercent = computed(() => {
    const s = this.systemStats.value;
    return s ? (s.memory_used / s.memory_total) * 100 : 0;
  });
  filteredProcesses = computed(() => {
    // [REQ-PRO-003] 搜索过滤逻辑
    const q = this.searchQuery.value.toLowerCase();
    return q ? this.processList.value.filter(p =>
      p.name.toLowerCase().includes(q)
    ) : this.processList.value;
  });
  searchQuery = signal<string>('');                 // [REQ-PRO-003]
  sortColumn = signal<SortColumn | null>(null);     // [REQ-PRO-002]
  sortDirection = signal<'asc' | 'desc'>('asc');
}

export const appState = new AppState(); // 单例导出
```

**ADR-5：为什么选择 Signals 而非 Framework？**（详见第七节）

#### 2.1.6 Persistence — 持久化管理器

```
职责：模块布局与用户设置的序列化/反序列化 [REQ-PST-001~002]

存储键（使用 tauri-plugin-store）:
  ┌──────────────────────────────────────────────────────────┐
  │ 键名                 │ 存储内容                           │
  ├──────────────────────┼───────────────────────────────────┤
  │ module-layouts       │ 所有模块的位置/尺寸/停靠/透明度/模糊 │
  │ app-settings         │ 全局设置（默认透明度/模糊等）        │
  │ custom-commands      │ 终端自定义指令 [REQ-TERM-005]      │
  └──────────────────────────────────────────────────────────┘

API:
  save(key: string, data: any): Promise<void>
  load<T>(key: string): Promise<T | null>
  remove(key: string): Promise<void>
  exportConfig(): Promise<string>         // [REQ-PST-003] 后期
  importConfig(json: string): Promise<void> // [REQ-PST-003] 后期
```

**持久化触发时机：**
- 拖拽结束 → 保存 `module-layouts`（防抖 500ms）
- 缩放结束 → 同上
- 贴靠/解锁 → 同上
- 透明度/模糊滑块松开 → 保存 `app-settings`
- 自定义指令变更 → 保存 `custom-commands`
- 应用退出前 → 全量保存

#### 2.2.2 模块注册约定

每个模块的 `index.ts` 默认导出一个 `ModuleDescriptor`：

```typescript
interface ModuleDescriptor {
  id: string;                    // 唯一标识，如 'icon-box'
  title: string;                 // 显示名称，如 '图标收纳盒'
  defaultState: Partial<ModuleState>;  // 默认位置和尺寸
  create(): ModuleBase;          // 工厂方法
}
```

`main.ts` 中的注册入口：

```typescript
import { IconBoxDescriptor } from './modules/IconBox';
import { MonitorDescriptor } from './modules/Monitor';
import { ProcessDescriptor } from './modules/Process';
import { TerminalDescriptor } from './modules/Terminal';

moduleManager.register(IconBoxDescriptor);
moduleManager.register(MonitorDescriptor);
moduleManager.register(ProcessDescriptor);
moduleManager.register(TerminalDescriptor);
moduleManager.initAll();
```

#### 2.2.3 图标收纳盒模块（REQ-ICON-001~007）

| 架构组件 | 职责 | 对应需求 |
|----------|------|----------|
| `IconBox.ts` | 继承 ModuleBase，管理图标数据与交互协调 | REQ-ICON-001~007 |
| `IconGrid.ts` | CSS Grid 网格渲染引擎，支持每行 6-12 图标 | REQ-ICON-002,003 |
| `IconItem.ts` | 单个图标渲染（图标 + 文件名，换行省略） | REQ-ICON-005 |
| `DragMirror.ts` | 拖拽镜像（半透明跟随副本） | REQ-ICON-006 |

**核心逻辑：**

```
Rust FilePoller
     │ 每 300ms 推送 Vec<FileEntry>
     ▼
前端 listen('desktop:files')
     │
     ├──→ 增量 diff（新增/删除/重命名）
     ├──→ StateManager 更新
     └──→ IconGrid 增量更新 DOM（不重建全部）

双击图标
     └──→ invoke('open_file', { path }) → Rust opener
```

**关键实现细节：**
- 仅主线程操作 DOM（无 Web Worker 必要，diff 计算极轻量）
- 增量更新时用 `DocumentFragment` 批量操作减少重排
- 图标缓存使用 `Map<path, HTMLElement>` 实现 O(1) 查找

#### 2.2.6 命令行终端模块（REQ-TERM-001~005）

| 架构组件 | 职责 | 对应需求 |
|----------|------|----------|
| `TerminalView.ts` | 继承 ModuleBase，xterm.js 容器 | REQ-TERM-001 |
| `CustomCommands.ts` | 快捷指令按钮组 | REQ-TERM-005 |

**核心逻辑：**

```
初始化流程：
  1. TerminalView.init()
  2. 创建 Terminal 实例 → 挂载到 contentArea [REQ-TERM-001]
  3. invoke('init_shell') → Rust 创建 cmd.exe 子进程 [REQ-TERM-002]
     ├── 返回 session_id
     └── std::thread::spawn 开始读取 stdout

输入流程：
  用户输入 → xterm.js onData → invoke('write_stdin', { session_id, data })
     └── Rust ShellManager.write_stdin() → child.stdin.write_all()

输出流程：
  Rust stdout thread → app_handle.emit('shell:output', { session_id, data })
     → 前端 listen → terminal.write(data)

彩色输出：xterm.js 原生支持 ANSI 转义序列 [REQ-TERM-004]

自定义指令：从 store 加载 → 渲染为快捷按钮 → 点击调用 write_stdin [REQ-TERM-005]
```

#### 模块显隐控制流 [REQ-SYS-003]

全局快捷键触发模块显隐的完整数据流：

```
用户按 Ctrl+Shift+D
    │
    ▼
tauri-plugin-global-shortcut (Rust)
    │
    ├──→ app.global_shortcut().on_shortcut() 触发
    │
    ▼
window.emit("app:toggle-modules")  ← 向 WebView 发送事件
    │
    ▼
前端 listen("app:toggle-modules")
    │
    ├──→ ModuleManager.toggleModules()
    │       │
    │       ├── 遍历所有模块实例
    │       ├── 切换 CSS display: none / 恢复
    │       └── 持久化显隐状态
    │
    └──→ 窗口背景始终透明可见，不受影响
```

**关键设计决策**：不直接调用 `window.hide()/show()`，因为：
1. 窗口背景需始终保持可见（透明层穿透到壁纸）
2. 模块显隐不应影响窗口 Z 序和焦点状态
3. 未来支持按模块 ID 独立控制显隐（配置项）

| 层级 | 方案 | 依赖 |

| 首选 | Tauri 2 内置 `set_always_on_bottom()` + `set_skip_taskbar()` + `set_focusable()` | 无 |
| 备选 | `SetWindowPos(HWND_BOTTOM)` + `SetWindowLong(WS_EX_NOACTIVATE)` | Win32 API |

```rust
// lib.rs 中的事件管道建立
fn setup_event_pipelines(app: &mut App) -> Result<()> {
    let app_handle = app.handle().clone();

    // [REQ-MON-001~003 / REQ-PRO-001] 系统监控 + 进程列表

    // [REQ-ICON-001] 桌面文件轮询

    Ok(())
}

fn setup_system_monitor(app_handle: AppHandle) {
    std::thread::spawn(move || {
        let mut monitor = SystemMonitor::new(
            Duration::from_millis(DEFAULT_STATS_INTERVAL),    // 1000ms
            Duration::from_millis(DEFAULT_PROCESS_INTERVAL),  // 500ms
        );
        monitor.start();
        while monitor.is_running() {
            // 双通道接收
            cross_channel::select! {
                recv(monitor.stats_rx) -> stats => {
                    app_handle.emit("system:stats", &stats).ok();
                }
                recv(monitor.process_rx) -> processes => {
                    app_handle.emit("system:processes", &processes).ok();
                }
            }
        }
    });
}
```

## 五、设计模式全景

| 模式 | 使用位置 | 说明 | 对应需求 |
|------|----------|------|----------|
| **模板方法** | `ModuleBase` | 基类定义生命周期骨架，子类实现 `renderContent()` | 全模块 |
| **单例** | `ModuleManager`, `EventBus`, `AppState` | 全局唯一的核心管理对象 | REQ-FRM-004,005 |
| **注册模式** | `ModuleManager.register()` | 模块按描述符注册，统一管理 | 全模块 |
| **策略模式** | `DragEngine + DragStrategy` | 三种可互换的拖拽/贴靠/缩放行为 | REQ-FRM-001,002,003 |
| **中介者模式** | `EventBus` | 模块间解耦通信 | 跨模块交互 |
| **信号/观察者** | `@preact/signals` | 状态变化自动触发 UI 更新 | 全模块 |
| **门面模式** | `services/` 各服务 | 封装底层系统 API，提供简洁接口 | REQ-SYS, REQ-MON |
| **生产者-消费者** | 服务线程 → Tauri 事件 | 后台采集与前端消费完全解耦 | REQ-MON, REQ-ICON |
| **命令模式** | `#[tauri::command]` | 前端 invoke 调用入口统一封装 | 全部 invoke 需求 |
| **工厂方法** | `ModuleDescriptor.create()` | 模块实例的创建接口 | 全模块 |

### 模式使用原则

1. **唯一性**：单例模式仅用于 ModuleManager、EventBus、AppState 三个核心管理器，其余对象通过构造函数注入
2. **替代关系**：模块间通信**优先通过 EventBus**，严禁直接引用另一个模块的实例
3. **层级调用**：Core 层可调用 Utils，不可反向；Modules 层可调用 Core 和 Utils，不可在同级模块间直接调用
4. **数据流单向**：`Rust Service → Tauri Event → EventBus → StateManager → Module UI`，禁止模块直接修改其他模块的状态

## 七、架构决策记录 (ADR)

### ADR-1：为什么选 `@preact/signals` 而非完整框架？

| 方案 | 体积 | 框架耦合 | Reactivity | 结论 |
|------|------|----------|------------|------|
| React/Vue | ~40KB+ | 高 | 声明式但需 VDOM | ❌ 太重量级 |
| `@preact/signals` | ~1.5KB | 无 | 自动追踪，直接 DOM | ✅ **选择** |
| Valtio | ~4KB | 无 | Proxy-based | 可选但偏重 |
| 原生 EventTarget | 0 | 无 | 手动触发 | 代码冗长 |

**结论**：Signals 最符合"原生三件套"理念，仅增加响应式能力不引入框架。自动追踪依赖树，直接操作 DOM。

### ADR-2：为什么用 `transform` 而非 `left/top` 做拖拽？

- **性能**：`transform` 在 GPU 合成层执行，不触发 Layout/Paint；`left/top` 触发完整回流
- **子像素精度**：`transform` 支持子像素渲染，动画更平滑
- **代价**：`transform` 不影响文档流，需额外维护逻辑位置用于贴靠判定（拖拽结束回写 `position`）

### ADR-3：为什么桌面文件用轮询而非 `notify` crate？

| 方案 | 稳定性 | CPU 占用 | 实现复杂度 |
|------|--------|----------|------------|
| `ReadDirectoryChangesW` | 低（网络驱动/虚拟文件夹不稳定） | 低 | 高 |
| `notify` crate 事件驱动 | 中（部分桌面环境兼容问题） | 低 | 中 |
| **轮询 300ms** | 高 | ~1% | 低 |

**结论**：轮询在 Windows 桌面上更稳定且可预测。增量 diff 对比确保每次推送量极小。

### ADR-4：为什么终端初期用 `cmd.exe` 管道而非 `portable-pty`？

- **复杂度**：`portable-pty` 在 Windows 上需处理 ConPTY API，引入额外系统依赖
- **MVP 原则**：基础命令（`dir`, `cd`, `mkdir`, `rm`, `tasklist`）通过 `std::process::Command` 管道即可满足
- **限制**：不支持交互式命令（`ping -t`, `python` REPL）— 标记为 REQ-TERM-006（P3，后期）
- **升级路径**：后期可替换为 `portable-pty`（仅涉及 ShellManager 内部改造，前端接口不变）

### ADR-5：为什么系统监控和进程列表共享同一服务？

- **数据源相同**：`sysinfo::System` 一次 `refresh_all()` 即可同时获取 CPU/内存和进程数据
- **频率不同**：通过双通道（`stats_tx` / `process_tx`）和各自的时间计数器实现 1s / 0.5s 不同频推送
- **避免资源竞争**：单个服务实例 + 单一线程，无需加锁

### ADR-6：前端增量 DOM 更新的策略

- **图标列表**：`Map<filepath, HTMLElement>`（O(1) 查找），diff 后只操作变化的 DOM 节点
- **进程列表**：全量替换，因进程数通常不超过 300，0.5s 全量重绘开销 < 2ms
- **监控面板**：仅更新数字文本节点（`textContent`），不重建卡片 DOM
- **原则**：能用 `textContent` 更新就不动 DOM 结构

## 八、文件命名与代码规范

### 8.1 命名约定

| 类型 | 约定 | 示例 |
|------|------|------|
| 类/接口 | PascalCase | `ModuleBase`, `DragEngine` |
| 变量/函数 | camelCase | `activeModuleId`, `saveLayout()` |
| 文件 (导出类) | PascalCase | `ModuleBase.ts`, `DragEngine.ts` |
| 文件 (非导出工具) | camelCase | `tauriApi.ts`, `constants.ts` |
| Rust 文件 | snake_case | `system_monitor.rs` |
| Rust 类型 | PascalCase | `SystemMonitor`, `SystemStats` |
| Rust 函数/方法 | snake_case | `get_system_stats` |
| CSS 类名 | kebab-case | `.module-titlebar`, `.icon-grid` |
| CSS 变量 | kebab-case | `--blur-strength`, `--bg-opacity` |
| Tauri 事件 | 命名空间:事件 | `system:stats`, `desktop:files` |

### 8.2 代码风格约定

- **TypeScript**: 严格模式 (`strict: true`)，显式类型注解优先于类型推断
- **Rust**: 遵循 `rustfmt` + clippy，所有公有函数必须有 doc comment（`///`）
- **前端 DOM**: 使用 `el.dataset.*` 存取数据属性，避免自定义属性污染
- **CSS 变量**: 声明在 `:root` 中，模块级覆盖在各自类作用域内
- **注释**：解释"为什么"而非"是什么"（后者代码自身说明）
