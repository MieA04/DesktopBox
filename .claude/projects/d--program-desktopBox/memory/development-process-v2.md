# 多智能体研发交付流程 v2

2026-06-01 将原有 PM → Dev → QA 三角色流程升级为六角色流程：

| 角色 | 原角色 | 拆分说明 |
|------|--------|----------|
| **PM（项目经理）** | PM | 不变，继续负责需求拆解和计划制定 |
| **技术总监** | 从PM拆分 | 新增。负责评判需求可行性、技术框架选型、技术方案评审、代码架构规划、代码规范制定 |
| **前端开发** | 从Dev拆分 | TypeScript/HTML/CSS 前端实现 |
| **后端开发** | 从Dev拆分 | Rust 后端实现 |
| **业务审查** | 从QA拆分 | 审查功能是否符合业务要求。重点关注：功能与计划的一致性、快捷键等功能触发是否正确 |
| **代码审查** | 从QA拆分 | 审查代码质量、架构规范遵守、代码规范执行、安全问题 |

## 执行顺序

PM → 技术总监 → 前端开发+后端开发(并行) → 业务审查 → 代码审查

## 返工闭环

- 业务审查 REJECT → 直接转发给对应开发 → 重新业务审查
- 代码审查 REJECT → 转发技术总监评估分配 → 对应开发修复 → 重新代码审查+业务审查

## 对应文件

- `CLAUDE.md` — 工作流程和团队角色说明
- `.claude/skills/multi-agent-dev-team/SKILL.md` — 完整技能定义

**Why:** v1 的三角色流程中 PM 既做需求又做架构决策，Dev 同时处理前端和后端，QA 同时做功能验证和代码审查，导致职责边界模糊、返工路径不清晰。v2 拆分后每个角色职责聚焦，返工闭环路径明确。

**How to apply:** 需要调用 `/multi-agent-dev-team` 技能时，按 SKILL.md 中定义的六步骤顺序执行，在 Step 2 使用 `Plan` 类型调用技术总监，Step 3 并行调用两个 `developer`。
