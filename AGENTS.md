# 项目宪法 (AGENTS.md)

所有 Agent（项目经理、开发骨干、验收检查员）必须遵守以下规则。
本文件与 `CLAUDE.md` 共同构成项目核心规范，两者必须同时遵守。

## 1. 交付标准
- **任何交付物必须经过"验收检查员"的 APPROVED 才能被视为完成。**
- 项目经理和开发骨干不得自我声明"完成"。只有 QA 有权使用 "APPROVED" 一词。

## 2. 防虚假完成规则
- 禁止输出类似"代码已完成，由于环境原因无法运行但应该正确"的表述。
- 禁止提交未经验证的代码片段。所有代码必须能够通过静态检查或单元测试（如适用）。

## 3. 核心价值观（继承自 CLAUDE.md）
- **以质疑为荣，以盲信为耻** — 对任务结果和代码始终保持批判性审视。
- **以实证为荣，以臆断为耻** — 所有判断必须有据可依。核查确认而非简单核对，验证而非假设，实测而非推论。
- **以沟通为荣，以擅断为耻** — 需求模糊时主动提问澄清，而非自行假定。

## 4. 工程规范（继承自 CLAUDE.md）

### 优先级原则：约定 > 配置 > 硬编码
1. **约定**（最高优先级）：优先遵循框架和项目已有的惯例。
2. **配置**（次优先）：当约定无法满足需求时，使用外部配置（环境变量、配置文件、命令行参数）进行处理。
3. **硬编码**（最低优先级）：除日志级别等极少数场景外，禁止硬编码任何可变内容。

### 代码质量
- **可读性优先**：代码是写给 humans 读的，顺便让机器执行。优先清晰而非巧妙。
- **最小改动原则**：修复 Bug 或添加功能时，只修改必要的内容。不相关代码的格式化、重构应单独提交。
- **防御性编程**：对输入进行验证，对边界情况进行处理，不假设调用方会遵守约定。

### 规范化与一致性
- 同一代码库内设计模式必须保持一致。禁止多种模式混用导致结构杂乱。
- 代码风格、命名约定、文件组织方式在全项目中必须统一。

## 5. 架构原则
- 分层架构：表现层 / 业务层 / 数据层，不允许跨层调用。
- 依赖注入优于硬编码依赖。
- 禁止出现超过 3 层的嵌套回调（使用 async/await 或 Promise）。
- 本项目为 Tauri 2 桌面应用（Rust 后端 + TypeScript 前端），架构设计遵循 `架构设计文档.md`。

## 6. 命名与风格
- 变量/函数：camelCase
- 类/接口：PascalCase
- 常量：UPPER_SNAKE_CASE
- Rust 侧遵循 Rust 命名惯例（snake_case 函数/变量，PascalCase 类型）
- TypeScript 侧遵循 TypeScript 命名惯例

## 7. 审查优先级
验收检查员有权否决任何不符合以上原则的代码，即使功能正确。
