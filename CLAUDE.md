# DesktopBox 项目核心开发守则

## 🏗 项目概览
- **项目名称**：DesktopBox — 桌面图标管理与系统监控辅助工具
- **目标平台**：Windows 10
- **技术栈**：Tauri 2 + TypeScript + 原生 HTML/CSS/JS + Vite
- **项目目录**：`d:\program\desktopBox`

---

## ⚖️ 核心开发守则（AI 开发守则）

### 三荣三耻
- **以质疑代码为荣，以信任任务报告为耻**
- **以核查确认为荣，以简单核对为耻**
- **以询问用户为荣，以模糊需求自主决定为耻**

### 质疑精神
- 时刻对任务结果与代码保持质疑，AI 存在幻觉是事实，必须时刻质疑工作成果，防止幻觉导致任务虚假完成
- 时刻对结论保持质疑，错误往往发生在看似正确的结果中

### 工程要求
- **工程必须规范**：所有代码与结构在开始开发前就必须进行规范制定
- **执行任务必须遵循规范**：编写代码必须符合代码规范，即使只是修改一个条件变量也必须做到遵循代码规范，即使规范会导致代码冗长
- **设计模式保持一致性**：所有代码层级的设计模式都必须保持一致，禁止多种设计模式混用导致代码结构杂乱
- **约定 > 配置 > 硬编码**：在任何情况下，都先考虑约定和配置，除了日志外任何内容都不应该使用硬编码

## 📐 技术规范

### 技术栈
| 层级 | 选型 |
|------|------|
| 前端核心 | 原生 TypeScript + HTML/CSS（无虚拟 DOM） |
| 构建工具 | Vite |
| 桌面框架 | Tauri 2 (Rust 后端) |
| 窗口与透明 | Tauri 原生 `window.setEffects` / WinAPI fallback / CSS `backdrop-filter: blur` 降级 |
| 系统监控 | `sysinfo` crate (Rust) |
| 进程列表 | `sysinfo` + `winapi` |
| 文件系统监控 | `notify` crate (Rust) 或轮询 |
| 终端集成 | `xterm.js` + Rust 伪终端（`cmd.exe` 桥接） |
| 持久化 | `serde` + `tauri-plugin-store` |
| 状态管理 | `@preact/signals` 或 `valtio`（轻量响应式）|

### 架构原则
- 模块化设计（ES Modules），TypeScript 严格模式
- Rust 后端负责系统级操作（WinAPI、文件监控、进程管理）
- 前端负责 UI 渲染与交互，通过 Tauri 事件/命令与后端通信
- 布局与用户设置持久化为 JSON 格式

## 🔄 工作流程约定

1. **开始新任务前**：先通读本项目文件（CLAUDE.md）确认规范
2. **编码前**：明确设计模式与代码结构规范，与用户确认后再开始
3. **每次修改**：遵循已有代码的风格与模式，保持一致性
4. **任务完成后**：质疑成果、核查确认，不盲目信任自检结果
5. **需求不明确时**：先询问用户，不自主决定模糊需求
